Je contrôle la doc Vercel sur trois points précis : racine du projet, outputDirectory et comportement des rewrites SPA.
Thought for 32s
Oui. À ce stade, la config Replit est cohérente avec la doc Vercel. Vercel indique bien que vercel.json doit être placé à la racine du projet, et que "$schema": "https://openapi.vercel.sh/vercel.json" sert à l’autocomplétion et à la validation du fichier.
Le point décisif est celui-ci : sur Vercel, le Root Directory est par défaut vide, ce qui correspond à la racine du repo. Vercel précise aussi que outputDirectory est interprété relativement à la racine du projet Vercel. Donc, avec Root Directory vide, ton outputDirectory: "artifacts/metabo-ai-website/dist/public" est logique. Si tu changes plus tard le Root Directory vers artifacts/metabo-ai-website, cette valeur deviendra fausse et devra passer à dist/public.

Pour une app Vite en mode SPA, Vercel documente bien qu’il faut une rewrite vers index.html pour que les routes profondes fonctionnent. Leur exemple officiel utilise "/(.*)" -> "/index.html". Ta version avec "/((?!.*\\.).*)" est plus restrictive, mais elle reste dans la logique d’une SPA qui veut laisser passer les assets statiques sans les réécrire.

La conclusion opérationnelle est simple : si tu importes le repo sur Vercel sans définir de Root Directory, le vercel.json actuel est le bon candidat de prod. En revanche, si tu sélectionnes un sous-dossier comme Root Directory, il faudra réaligner buildCommand, outputDirectory et l’emplacement du vercel.json sur cette nouvelle racine de projet.

Ton fichier final peut donc être considéré comme validé :

{
  "$schema": "https://openapi.vercel.sh/vercel.json",
  "framework": null,
  "installCommand": "pnpm install",
  "buildCommand": "pnpm --filter @workspace/metabo-ai-website run build",
  "outputDirectory": "artifacts/metabo-ai-website/dist/public",
  "rewrites": [
    {
      "source": "/((?!.*\\.).*)",
      "destination": "/index.html"
    }
  ]
}
Le bon move maintenant : sur Vercel, laisse Root Directory vide, fais le premier déploiement, puis teste directement /privacy, /terms, /support et /legal. Si une 404 apparaît malgré un build vert, le premier suspect sera presque toujours un décalage entre Root Directory et Output Directory.
